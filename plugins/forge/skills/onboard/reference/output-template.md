# onboard.md Output Template

This file defines the exact structure for `.forge/context/onboard.md`.
Referenced by `/forge:onboard` Step 11.

## Design principles

- **Navigation, not inventory.** Every section answers a specific question
  AI developers will ask when working on the codebase.
- **Observations, not rules.** Rules live in calibrate's output
  (`architecture.md`, `constraints.md`, `testing.md`, `conventions.md`).
- **Section markers enable incremental updates.** Each section is wrapped
  in HTML comments with `verified=<commit-hash>` so subsequent runs can
  diff and refresh only changed sections.
- **Confidence tags everywhere.** Every factual claim carries `[code]`,
  `[config]`, `[build]`, `[readme]`, or `[inferred]`.
- **Preserve blocks for human additions.** Content inside
  `<!-- forge:onboard:preserve -->` is carried across all regenerations.

## Example domain for illustrations

All concrete examples below use a **generic e-commerce order platform**
(domain classes like `Order`, `Customer`, `Payment`; packages like
`com.example.shop.*`; service name `order-service`). These are
illustrative only — replace with the actual identifiers found in the
target project.

## 9. Known Traps

> Environment and history traps that are hard to see from the code alone.
> Code-level anti-patterns and forbidden patterns are NOT listed here.

### Environment

- **Internal CA cert must be imported locally:** JVM may not trust
  company-internal or region-specific domains by default. Import the
  required certificates from `./config/` into
  `$JAVA_HOME/jre/lib/security/cacerts`. The Dockerfile typically
  handles this in production. [code]
- **Apple Silicon WireMock ALPN failure** (if WireMock version < 3.x):
  symptom `No Server ALPNProcessors!`. Workaround: export
  `WIREMOCK_SERVER_HTTPS_PORT=-1` and
  `WIREMOCK_SERVER_HTTPS_PORT_DYNAMIC=false`. [inferred]
- **VPN + Nexus credentials required for build:** `repoUser` /
  `repoPassword` in `~/.gradle/gradle.properties`; without them, Gradle
  can't resolve internal `com.example.internal:*` starters. [build]

### History / legacy

- **README version drift:** README claims Java 8 + MySQL 5.7. Actual
  versions: Java 11 (`gradle.properties: javaVersion=11`) + MySQL 8
  (`mysql-connector-java:8.0.26`). Treat the README infrastructure
  section as stale. [build] [conflict]
- **Mixed URL versioning:** newer endpoints use `/api/v2/`, older
  endpoints (catalog, imports, reporting) use `/api/`. Both run in the
  same context. [code]
- **`fakestub/` is a stub, not the real integration:** code under
  `{module}/adapter/api/fakestub/` returns canned responses for local
  testing. The real integration lives in `{module}/adapter/{vendor}/`.
  [code]
- **`@Ignore` vs `@Disabled` mix-ups observed:** JUnit 4's `@Ignore` does
  not disable JUnit 5 tests. Historical commits have mistakenly used
  `@Ignore` on `@Test` methods that still run. [code]

### Git hook side effect

- **`initGitHooks` task in root `build.gradle` runs on every Gradle
  invocation** and silently configures `git config core.hooksPath
  .githooks`. Installs `.githooks/pre-push`. To disable: `git config
  --unset core.hooksPath`. [code]

### ShardingSphere local gotcha

- `db.sharding.enabled=true` is default. A plain single-datasource URL
  without matching ShardingSphere config will fail at startup. Local
  setup must match the sharding config in `application.yml`. [config]

### Feature toggle console exposure

- Togglz console at `/api/v2/{module}/togglz-console` with
  `togglz.console.secured=false`. [config]

> 反模式与硬性规则见 `.forge/context/constraints.md`（由 /forge:calibrate 产生）

## Section marker rules

Every section is wrapped in opening and closing HTML comments:

```markdown
<!-- forge:onboard section=<name> verified=<commit-hash> generated=<YYYY-MM-DD> -->
## N. Section Title

...

<!-- /forge:onboard section=<name> -->
```

- `<name>` is one of the 10 canonical names (9 content + `document-confidence`).
- `verified=<commit-hash>` is the short commit hash at the time of the
  last successful verification pass for that section.
- `generated=<YYYY-MM-DD>` is the date the section's body was last written
  (different from `verified=` only when the section was reused).

Incremental mode logic:
- Current HEAD == `verified=<hash>` → skip section entirely
- Current HEAD != `verified=<hash>` → rescan; if result differs, rewrite
  body + update both attributes; if identical, update `verified=` only

## Preserve block rules

Human-added content inside a section is protected by preserve markers:

```markdown
<!-- forge:onboard:preserve -->
{any content here is never rewritten by the skill}
<!-- /forge:onboard:preserve -->
```

- Preserve blocks may appear anywhere inside a section
- Multiple preserve blocks per section are allowed
- On regeneration, the skill extracts all preserve blocks, writes the new
  section body, then re-injects preserve blocks in their original
  positions (tracked by surrounding context)
- `--regenerate` does NOT bypass preserve blocks — they are sacred
