## 1. Project Identity

Forge is a **Claude Code plugin** that provides an AI-driven software
development workflow for existing (legacy) codebases [readme] [code]. It
decomposes the full feature-development cycle into nine interconnected
skills; each skill consumes structured Markdown artifacts from a shared
`.forge/` directory, performs its analysis, and writes new artifacts for
the next skill to read — forming a context chain that persists across
Claude Code sessions [code].

Primary users: software engineers working on existing code who want AI
assistance that respects project conventions, stays within scope, and
pauses to ask rather than assume [readme]. The design principle is
stated explicitly: **AI is the developer, humans provide intent and
judgment** [readme].

Forge is **meta-tooling**, not an application — no HTTP server, no
database, no runtime. It also **self-hosts**: this repository uses its
own skills to develop itself, so the `.forge/` directory here is a live
sample of what forge produces in a real target project [code].

**Runnable unit:** N/A — forge is activated by Claude Code via the
`/forge:*` slash commands (see Section 5: Entry Points) [code].
**Repo name vs plugin name:** repository `forge` (GitHub: `lotusp/forge`)
hosts a Claude Code marketplace whose single plugin is also named `forge`
and lives in `plugins/forge/` [build].

### Observed layering

Four layers observed in the repository [code]:

1. **Orchestrator** — `plugins/forge/skills/forge/` — the master
   `/forge:forge` skill runs `scripts/status.mjs` to detect state and
   routes to the correct sub-skill.
2. **Skills** — `plugins/forge/skills/{onboard,calibrate,clarify,design,
   tasking,code,inspect,test}/` — eight domain-specific skills, each
   with its own `SKILL.md` and optional `reference/` and `scripts/`
   sub-directories.
3. **Agents** — `plugins/forge/agents/{forge-explorer,forge-architect,
   forge-reviewer}.md` — three task-scoped agents invoked concurrently
   by specific skills (explorer ← clarify, architect ← design,
   reviewer ← inspect).
4. **Artifacts** — `.forge/context/` (project-wide) + `.forge/features/
   {slug}/` (per-feature) — the persistent context chain that
   connects skills across sessions.

Observed call direction: Orchestrator → Skill → (optional Agent) →
Artifact. Skills never invoke other skills directly — they write
artifacts that the next skill reads [code].

Observed routing authority: `scripts/status.mjs` is the sole authority
for the "next step" decision; the `/forge:forge` SKILL.md has an IRON
RULE forbidding the orchestrator from second-guessing the script's
output [code].

fails its Prerequisites check if its upstream artifacts are missing:

```
onboard → calibrate → clarify → design → tasking → code → inspect → test
```

The `forge` orchestrator's `status.mjs` detects the most advanced
artifact in `.forge/` and emits a `[ACTION]` line telling the user
which skill to invoke next [code].

## 5. Entry Points & Call Chains

> This project has no HTTP API, no CLI, no background jobs, and no
> message consumers. Entry points are **Claude Code slash commands**
> triggered by the user typing `/forge:{skill-name}`.

### Slash commands

Total commands: **9** [code]

| Command | Skill | Purpose | When to invoke |
|---------|-------|---------|----------------|
| `/forge:forge [arg]` | orchestrator | Auto-detect state and route | Any time (recommended primary entry) |
| `/forge:onboard [--regenerate \| --section=<n>]` | onboard | Generate this artifact | First time on a project |
| `/forge:calibrate` | calibrate | Extract conventions | After `onboard` |
| `/forge:clarify <requirement>` | clarify | Analyze a new requirement | When starting a feature |
| `/forge:design <slug>` | design | Produce technical design | After `clarify` |
| `/forge:tasking <slug>` | tasking | Break into ordered tasks | After `design` |
| `/forge:code <task-id>` | code | Implement one task | For each `T{NNN}` |
| `/forge:inspect <slug \| file>` | inspect | Review completed code | After code tasks complete |
| `/forge:test <slug>` | test | Generate tests | After `inspect` passes |

### Representative call chains

#### New-project onboarding
- Entry: user runs `/forge:forge` in a fresh repo [code]
- Orchestrator: `forge/SKILL.md` runs `scripts/status.mjs` [code]
- Script decision: no `.forge/context/onboard.md` → emits
  `[ACTION] skill=onboard` [code]
- Orchestrator routes to `/forge:onboard` [code]
- Outputs: `.forge/context/onboard.md` + JOURNAL entry [code]

#### Implement one task
- Entry: user runs `/forge:forge T005` [code]
- Orchestrator: `status.mjs --task T005` grep-scans
  `.forge/features/*/plan.md` for `T005` [code]
- Script decision: locates task, emits
  `[ACTION] skill=code arg=T005` [code]
- `code` reads plan, clarify, conventions, constraints [code]
- Outputs: edited source files + `T005-summary.md` [code]

#### Concurrent code review
- Entry: user runs `/forge:inspect {slug}` [code]
- `inspect` reads all `T*-summary.md` to collect changed files [code]
- For each file: spawn one `forge-reviewer` agent (red 🔴) concurrently [code]
- Each agent reads the file, filters findings with confidence <80 [code]
- Results aggregated into `inspect.md` [code]

### Background jobs, CLI, gRPC, event listeners

**None** — this project has no runtime [code]. Omitting these
sub-sections per template rules.

## 9. Known Traps

> Environment and history traps that are hard to see from the code alone.
> Code-level anti-patterns and forbidden patterns are NOT listed here.

### Plugin-cache / reload gotchas

- **`/plugin update forge` is a no-op unless `plugin.json` version bumps.**
  If you edit a SKILL.md but don't bump `plugins/forge/.claude-plugin/
  plugin.json`'s `version` field, the update command says "already latest"
  and ships stale content. Always bump (patch for fixes, minor for new
  features, major for breaking artifact format changes) [build].
- **Plugin cache lives at `~/.claude/plugins/cache/forge/forge/{version}/`.**
  Local edits to the repo do not propagate to installed versions; use
  `--plugin-dir` for development (see Section 8) or `/reload-plugins`
  after `/plugin update forge` [config].
- **`/reload-plugins` reports skill count — if it shows 0 skills, the
  plugin failed to resolve.** Common causes: circular URL in
  `marketplace.json`, missing `"source"` field, stale cache. See the
  related git history fixes in Section 7.

### Nested directory structure (marketplace requirement)

- Claude Code's marketplace model requires the plugin to live in a
  subdirectory referenced from the marketplace manifest. This repo has
  `.claude-plugin/marketplace.json` at the root pointing to
  `./plugins/forge`, which holds the actual plugin content. **Do not
  move `plugins/forge/` content up to the repo root** — it will break
  installation [code]. Prior fix attempt: `e746845`
  (`chore(plugin): restructure into plugins/forge/ subdirectory`).

### `allowed-tools` pitfalls

- **Every SKILL.md that writes an artifact must include `Write` in its
  `allowed-tools` frontmatter field.** Missing `Write` silently prevents
  the artifact from being created even though the skill's Process
  instructs it to. Prior fix: `d99ed4b` (added `Write` to 6 skills that
  lacked it). [code]
- **Do not set `agent: Explore` on a skill that needs to write.** The
  built-in `Explore` agent is read-only (no `Write`, no `Bash`). Prior
  fix: same commit `d99ed4b` removed `agent: Explore` from `onboard`.
  [code]

### Stale references after rename

- Two skills were renamed due to conflicts with Claude Code built-ins:
  `plan` → `tasking` (to avoid `/plan`) and `review` → `inspect` (to
  avoid `/review`). Even after `764fcc0`, later commits (`b86fe40`) had
  to fix remaining stale references in agent files. **When renaming a
  skill, grep the whole repo — SKILL.md, agent .md files, status.mjs,
  state-machine.md, all docs.** [code]

### Self-hosting trap (meta-project)

- `.forge/` in this repo is **not** an example — it's the live
  development context for forge itself. Modifying `.forge/context/
  onboard.md` manually will be overwritten on the next
  `/forge:onboard --regenerate` unless wrapped in a preserve block
  (`<!-- forge:onboard:preserve -->`) [code].

### No automated tests

- The repo has no JUnit/Jest/pytest equivalent. Changes to SKILL.md or
  agent files are not regression-testable programmatically; validation
  is manual per `.forge/context/testing.md`. When changing a skill's
  Process or IRON RULES, **run the skill on a representative target
  project** and inspect the produced artifact against the review
  baseline [code].

### Node scripts assume CWD = target project

- `status.mjs`, `check-prerequisites.mjs`, `save-scan-state.mjs`, and
  `validate-output.mjs` all read paths like `.forge/context/onboard.md`
  relative to the current working directory. They are designed to run
  from the target project root, not from forge's own repo root [code].

### Empty placeholder directories

- `docs/decisions/` and `docs/milestones/` are empty directories
  reserved for future ADRs and release notes, respectively [code]. Not
  a trap, but worth noting so scanners don't flag them as broken links.

> 反模式与硬性规则见 `.forge/context/constraints.md`（由 /forge:calibrate 产生）
