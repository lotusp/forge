## Open Design Space (from clarify)

**DI-1** — Evidence item internal representation schema  
Context: The redesign calls for a classification layer between scan and render. Design must decide what data structure carries a claim through the pipeline (claim text, confidence level, source tag, classification category, traceable evidence ref). Came up because the classification layer is internal-only — the user should not see intermediate data, but the design must stabilize the schema so it can be referenced throughout the SKILL.md process description.

**DI-2** — Claim classification mechanism stability  
Context: In long onboard runs (12+ profiles/dimensions), an LLM classification instruction at the top of SKILL.md can drift. Design must decide whether classification rules are embedded inline at each dimension/profile step, or in a standalone lookup table, or enforced via Iron Rule. Came up because the brief explicitly flags "how to stay stable under long prompts" as a key design question.

**DI-3** — Which profiles/dimensions to split, rewrite, or remove  
Context: `local-dev.md` directly produces operational content (Docker commands, env file copy). `architecture-layers.md` produces inferred patterns as rules without separation. The "What to avoid" sections in dimension templates blur recommended-pattern with enforced-rule. Design must enumerate which files change and how. Came up from code inspection — the profile template structures need surgical redesign.

**DI-4** — Deterministic extraction helpers  
Context: The brief requires that count-based and route-based claims are deterministic. Current implementation uses approximate language ("~6 TODO markers"). Design must decide which extraction operations get promoted to `scripts/*.mjs` helpers and what their output contracts are. Came up from evidence-discipline section of the brief (count-based claims, route-based claims must be deterministic).

**DI-5** — Change Navigation Map generation algorithm  
Context: This section must be derived from "real code relationships" without becoming speculative. Design must decide: what signals to extract (import graphs, test file co-location, event names, route-to-service mapping), how to derive the map per kind, and how to express it in markdown without the entries becoming stale. Came up because the brief describes the section's purpose but not its generation method.

**DI-6** — Section template redesigns for architecture.md and constraints.md  
Context: The brief specifies three-section architecture.md (Observed Structure / Enforced Rules / Recommended Direction) and three-section constraints.md (Hard Constraints / Process / Quality Gates / Current Business Caveats). Current dimension templates (architecture-layers.md, hard-constraints.md, anti-patterns.md) produce a different shape. Design must produce new template markup for all affected dimension files. **DI-7** — Incremental mode interaction with claim classification  
Context: Claim classification changes the semantic meaning of each section body. A section from a pre-redesign run may have "What to avoid" mixing enforced rules and recommendations. Incremental mode's dirty-detection (body-signature hash) will flag these as dirty and rewrite them — but the design must confirm this is the intended upgrade path and that the merge algorithm handles reclassified content correctly. Came up because backwards compatibility is a [WHAT] question answered in Q&A, but incremental mode's behavior under reclassification is an implementation detail.
